## 주요 담당 업무

### 1. 프로젝트 메타데이터 자동 분석
- 프로젝트 설명문, README, 코드 파일을 분석하여 핵심 정보를 추출
- 기술 스택, 프로젝트 주제, 핵심 키워드, 사용 언어 등을 구조화
- 추출된 메타데이터를 검색 및 추천 기능에 활용할 수 있도록 정리

### 2. 자연어 기반 검색
- 사용자의 자연어 질문에서 의도와 조건을 해석
- 질문 속 도메인, 기술 스택, 목적, 난이도 등의 정보를 추출
- 추출된 조건을 기반으로 관련 프로젝트를 검색할 수 있는 구조 설계

### 3. 추천 기능
- 검색 결과 및 메타데이터를 바탕으로 유사 프로젝트 추천
- 공통 기술 스택, 유사 주제, 난이도 등을 기준으로 추천 결과 생성
- 추천 이유를 함께 제공할 수 있도록 출력 구조 설계

### 4. 시맨틱 검색 최적화
- 프로젝트 설명 및 문서를 임베딩으로 변환
- 벡터 기반 유사도 검색 구조를 검토
- 키워드 검색과 의미 기반 검색을 결합한 탐색 방식 설계

### 5. 백엔드 연동
- AI 분석 결과를 백엔드 서비스에서 활용할 수 있도록 연동 구조 검토
- 검색 결과, 추천 결과, 메타데이터를 API 형태로 제공할 수 있도록 설계
- 백엔드 담당과 협업하여 서비스 적용 흐름 정리

다음은 본 서비스에서 처리하고자 하는 대표적인 자연어 질의 예시이다.

- 헬스케어 분야에서 이미지 기반 AI 프로젝트 예시 추천해줘

- 딥러닝 초보자가 참고하기 좋은 프로젝트 5개 추천해줘
- Unity랑 MediaPipe 조합으로 만든 선배 프로젝트 있나?
- 캡스톤 프로젝트 중에서 추천 시스템 관련 사례 보여줘

### 질의 해석 예시
| 자연어 질의 | 해석 조건 |

# 설계 시 고려 사항

2주차 작업을 진행하면서 다음과 같은 고려 사항을 정리하였다.

## 1. 단일 입력 출처만으로는 충분하지 않음
프로젝트 설명문이 짧거나 부정확할 수 있으므로,  
README, 보고서, 코드 구조를 함께 참고해야 한다.

## 2. 추출 결과의 정규화가 필요함
예를 들어 사용자가 다음과 같이 서로 다르게 적을 수 있다.

- JS / JavaScript
- Spring / Spring Boot
- AI / Artificial Intelligence

이러한 표현은 내부적으로 통일된 값으로 정규화해야  
검색과 추천 품질을 높일 수 있다.

## 3. 확정 정보와 추정 정보를 구분할 필요가 있음
일부 메타데이터는 문서에 명시적으로 존재하지만,  
일부는 코드 구조나 표현을 바탕으로 추정해야 한다.  
따라서 향후 구현 시에는 신뢰도 개념을 둘 수 있다고 판단하였다.

## 4. 검색 친화적인 구조가 필요함
메타데이터는 단순 저장용이 아니라  
키워드 검색, 필터링, 추천, 시맨틱 검색에 모두 활용되어야 한다.

# 자연어 질의 처리 구조

사용자는 단순 키워드가 아니라 문장 형태로 프로젝트를 찾고자 할 가능성이 높다.  
따라서 AI 모듈은 사용자의 질문을 다음과 같은 단계로 처리해야 한다고 정리하였다.

## 자연어 질의 처리 단계

1. 사용자 질문 입력
2. 질문에서 핵심 조건 추출
   - 프로젝트 주제
   - 기술 스택
   - 목적
   - 난이도
   - 결과 개수
3. 조건을 구조화된 검색 조건으로 변환
4. 검색 엔진 및 메타데이터 DB에서 후보 프로젝트 조회
5. 후보 결과를 정렬 및 재선별
6. 최종 추천 결과와 요약 정보 생성

|---|---|
| 헬스케어 분야 AI 프로젝트 추천해줘 | 주제=헬스케어, 개념=AI |

| Unity와 MediaPipe를 사용한 사례가 있나? | 기술 스택=Unity, MediaPipe |
| 추천 시스템 관련 캡스톤 프로젝트 찾아줘 | 프로젝트 유형=캡스톤, 주제=추천 시스템 |

이와 같이 자연어 질의를 구조화된 검색 조건으로 바꾸는 과정이  
AI 검색 모듈의 핵심 역할이라고 판단하였다.

# 검색 방식 설계

이번 주차에는 검색 기능을 단일 방식으로 처리하지 않고,  
다음과 같이 여러 검색 방식을 조합하는 구조를 검토하였다.

## 1. 키워드 검색
- 프로젝트명, 한 줄 소개, 상세 설명, README의 텍스트를 기반으로 검색
- 명시적으로 입력된 단어를 빠르게 찾는 데 유리
- 예: "React", "Spring", "헬스케어"

## 2. 메타데이터 필터 검색
- 기술 스택, 연도, 학기, 과목명, 프로젝트 유형 등 구조화된 정보를 기반으로 검색
- 사용자가 원하는 조건을 명확히 반영할 수 있음
- 예: 기술 스택 = React + Spring, 유형 = 캡스톤

## 3. 벡터 기반 시맨틱 검색
- 프로젝트 설명, README, 문서 내용을 임베딩으로 변환하여 의미 기반 유사도 검색 수행
- 문장 표현이 달라도 의미가 비슷한 프로젝트를 찾을 수 있음
- 예: "처음 배우기 좋은 AI 프로젝트" ↔ "입문자용 머신러닝 프로젝트"

## 4. 혼합 검색
- 키워드 검색 + 메타데이터 필터 + 벡터 검색을 함께 활용
- 검색 품질과 유연성을 동시에 확보할 수 있음
- 최종적으로는 혼합 검색 구조가 가장 적합하다고 정리하였다

# RAG 기반 검색 구조 초안

이 프로젝트는 단순 검색 결과 목록 제공을 넘어서  
사용자 질문에 맞는 프로젝트를 **설명과 함께 추천**하는 방향을 목표로 한다.  
이를 위해 RAG 기반 구조를 다음과 같이 정리하였다. :contentReference[oaicite:1]{index=1}

## RAG 처리 흐름

1. 사용자 질문 입력
2. LLM이 질문의 의도와 조건 분석
3. 분석된 조건을 기반으로 검색 수행
   - 메타데이터 검색
   - 키워드 검색
   - 벡터 검색
4. 관련도가 높은 프로젝트 후보 수집
5. 후보 프로젝트의 핵심 정보 정리
   - 프로젝트명
   - 기술 스택
   - 주제
   - 요약 설명
6. LLM이 검색 결과를 기반으로 응답 생성
   - 추천 프로젝트 제시
   - 추천 이유 설명
   - 필요 시 비교 및 요약 제공

# 추천 기능 구조 설계

검색 결과를 단순 나열하는 것만으로는 학습 지원 효과가 부족하다고 판단하였다.  
따라서 추천 기능은 다음과 같은 기준으로 구성하는 방향을 정리하였다.

## 추천 기준

| 추천 기준 | 설명 |
|---|---|
| 기술 스택 유사성 | 동일하거나 유사한 기술 스택을 사용하는 프로젝트 |
| 주제 유사성 | 같은 도메인 또는 유사 문제를 다루는 프로젝트 |
| 난이도 적합성 | 사용자의 수준에 맞는 프로젝트 |
| 프로젝트 유형 유사성 | 캡스톤, 수업 과제, 팀 프로젝트 등 맥락이 비슷한 사례 |
| 문서 의미 유사성 | 설명문, README, 보고서의 의미적 유사성 |

## 추천 결과 예시

추천 결과에는 다음 정보가 함께 제공되어야 한다고 정리하였다.

- 프로젝트명
- 한 줄 설명
- 주요 기술 스택
- 추천 이유
- 관련도 또는 공통 요소

예를 들어:

- **프로젝트 A**  
  - 기술 스택: React, Spring Boot  
  - 추천 이유: 동일한 프론트/백엔드 구조를 사용한 웹 프로젝트

- **프로젝트 B**  
  - 기술 스택: PyTorch, OpenCV  
  - 추천 이유: 헬스케어 이미지 처리라는 주제가 유사함

# 설계 시 고려 사항

## 1. 검색과 추천은 분리하되 연결되어야 함
검색은 후보를 찾는 과정이고, 추천은 그중 적절한 프로젝트를 다시 선별해 제시하는 과정이므로  
둘을 분리해서 설계하되 하나의 흐름으로 연결되도록 해야 한다.

## 2. 단순 키워드 일치만으로는 부족함
사용자의 질문은 다양하게 표현될 수 있으므로  
벡터 검색과 의미 기반 유사도 계산이 필요하다.

## 3. 추천 이유 제공이 중요함
사용자 입장에서는 왜 이 프로젝트가 추천되었는지 이해할 수 있어야 하므로  
결과와 함께 추천 근거를 제공해야 한다.

## 4. 백엔드와의 연동을 고려한 구조가 필요함
AI 기능은 독립적으로 끝나는 것이 아니라  
백엔드 API와 연결되어 서비스 화면에 결과를 전달해야 하므로  
응답 형태를 구조적으로 설계해야 한다.
